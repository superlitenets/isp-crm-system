It is NOT:

Wrong port

Wrong endpoint

Wrong auth

Wrong VLAN

Wrong provision

It is this:

CURLE_GOT_NOTHING (52)


And the cause is:

You are sending connection_request with timeout=15000
GenieACS tries to reach the ONU immediately.
If ONU is:

Behind NAT

Temporarily offline

ConnectionRequestURL unreachable

Firewall blocking

GenieACS waits → fails → closes socket → NBI drops connection → curl sees “Empty reply from server”.

So your PHP thinks task queue failed — but actually NBI closed during connection_request attempt.

🧠 What’s Really Happening

Your flow:

Upload provision ✔

Queue task + connection_request in same call ❌

GenieACS:

Queues task

Tries to contact device

Device unreachable

ConnectionRequest times out

NBI closes connection abruptly

Curl receives empty reply

PHP thinks task queue failed

But task may actually be queued.

✅ Correct Architecture (Professional Way)

Always:

STEP 1 — Queue task only
POST /devices/<id>/tasks
{
  "name": "provision",
  "provision": "wifi_bridge_config",
  "arguments": { ... }
}


NO connection_request parameter.

Wait for HTTP 200.

STEP 2 — Send connection_request separately
POST /devices/<id>/tasks
{
  "name": "connection_request"
}


And do NOT depend on its result.

If it fails — task remains queued.

Device will execute on next Inform anyway.

🎯 Why This Is Best Practice

GenieACS is event-driven.

You do NOT need connection_request if:

Inform interval = 300 seconds

Or device is online

Connection request is just acceleration.

Never make provisioning dependent on it.

🔥 Implementation Change

In your executeWifiBridgeConfig():

Remove this:

.../devices/<id>/tasks?connection_request


Instead:

// 1️⃣ Queue provision task
POST /devices/<id>/tasks
{
  "name": "provision",
  "provision": "wifi_bridge_config",
  "arguments": {...}
}

// 2️⃣ Fire connection request separately
POST /devices/<id>/tasks
{
  "name": "connection_request"
}


Wrap connection_request in try/catch and ignore failure.

🧠 Even Better

If your inform interval is 300 seconds (5 minutes):

You can skip connection_request entirely.

Device will pick it up on next inform.

For 5000 ONUs this is actually more stable.
