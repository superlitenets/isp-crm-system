That means:

Your provision script is referencing a TR-069 path that does NOT exist on the device.

Not an API issue.
Not a connection issue.
Not a VLAN issue.

It’s a wrong parameter path.

🔍 Where The Problem Is

From your logs:

Queuing bridge config on:
InternetGatewayDevice.WANDevice.1.WANConnectionDevice.2.WANIPConnection.1


And:

Queuing WiFi config on:
InternetGatewayDevice.LANDevice.1.WLANConfiguration.4


One of those paths does not exist on the ONT.

🚨 Very Likely Problem

HG8546M usually has:

WLANConfiguration.1
WLANConfiguration.2


Sometimes:

.3 and .4


But not always.

Your script assumes:

WLANConfiguration.4


If instance 4 does not exist → GenieACS throws:

Invalid parameter path at instances

✅ First Thing To Check

Open that device in GenieACS.

Expand:

InternetGatewayDevice.LANDevice.1.WLANConfiguration


Count how many instances exist.

You will likely see:

2


If only 1 and 2 exist → index 4 is invalid.

🔥 Second Possible Problem

This part:

WANConnectionDevice.2.WANIPConnection.1


You predicted new index = 2.

But AddObject does NOT immediately create instance when device is offline.

So path:

WANConnectionDevice.2


Does not exist yet.

GenieACS cannot reference predicted index before device actually creates it.

That is the classic mistake.

🧠 Why This Fails

You are doing:

addObject WANConnectionDevice

Predict index = 2

Immediately queue config using that predicted index

Device hasn’t confirmed instance creation yet

GenieACS cannot validate path

→ Invalid parameter path

🎯 Correct Way To Do This In GenieACS

You CANNOT safely predict instance numbers.

Instead you must:

Use declare with dynamic path handling.

Example:

var wan = declare("InternetGatewayDevice.WANDevice.1.WANConnectionDevice.", {path: 1});
var ip = declare(wan.path + "WANIPConnection.", {path: 1});

declare(ip.path + "Enable", null, {value: true});
declare(ip.path + "ConnectionType", null, {value: "IP_Bridged"});
declare(ip.path + "X_HW_VLAN", null, {value: 660});


Notice:

We NEVER hardcode .2
We use wan.path

That guarantees correct instance.

🔥 Same For WLAN

Instead of:

WLANConfiguration.4


Do:

var wlan = declare("InternetGatewayDevice.LANDevice.1.WLANConfiguration.", null, {path: 1});


Or better:

Find existing instance and modify it.

🏆 Root Cause Summary

Your PHP logic:

✔ Predicts WAN index
✔ Hardcodes instance
❌ But device hasn’t created it yet

GenieACS refuses invalid path.

💡 Immediate Fix

Stop predicting instance numbers in PHP.

Move entire logic into ONE GenieACS provision script.

Let GenieACS handle:

AddObject

Dynamic instance resolution

Path chaining

That eliminates 100% of these errors.
